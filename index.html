Final Project Report

Abstract

In this project, we simulated a piece of cloth burning, the color of the cloth changing accordingly, and a colormap of the tension felt at each piece of the cloth. Pulling equations and ideas from a research paper we found, we implemented a technique to make an anisotropic cloth, a discretized algorithm for heat transfer, and a new shader file to represent the changing state of the cloth. Each change in state has a different threshold and we use the final state to help determine the cloth’s willingness to fracture.

Technical Approach

Project Scope: Our original project involved simulating 3D solid melting as a result of heat transfer. We later decided to scale back our scope by changing the 3D block to a 2D sheet and removing the phase change aspect to simulate, instead, the burning of paper. Since we still wanted to maintain some rendering of heat transfer, so our project simplified down to heat input into a 2D sheet which after some research and consideration, is essentially observing the heat transfer as a piece of cloth burns.

Determining Paper Burnout

Each PointMass contains a burn_state (0 or not burning, 1 for burning, 2 for burnt) and an init_temperature. The PointMass changes its burn_state in correspondence to how much its temperature has increased from init_temperature.  Burn_state is then used to tell the shader what color to shade in the PointMass depending on its burn state. Each PointMass also stores a vector containing all the springs connected to it, which allows for convenient access and pointer access when dealing with the fiber fracturing algorithm.

Representing Fibers as Springs

We modified the original PointMass and Cloth structs so that each type of spring has an independent spring constant associated with it (ks_structural, ks_shearing, ks_bending), allowing us to adjust the properties of the cloth to more realistically represent paper. These can each be adjusted in the GUI.

We applied a set random offset for structural springs as such: each row of horizontal structural springs has their spring constant adjusted by a set proportion (10%) throughout the simulation per row, and each row of vertical springs has their spring constant adjusted by a set proportion (10%) per row throughout the simulation. This helps make our paper simulation more anisotropic to make for more realistic crumpling of the paper as it burns.

Other constants we set for the heat transfer algorithm include specific_heat_capacity, grammage, and dimensional_stability. The latter two can be adjusted in the GUI.

Innovative Changes

Whereas the paper we referenced implemented a particle-driven to kick off the simulation, we started our reaction with an initial flux approach. This means that instead of particles shooting heat onto the paper, we directly apply heat to a set of point masses (a corner or the whole bottom row, for example) and allow the heat to propagate. We can control the amount of heat and the amount of timesteps we inject the heat.

The original paper details a shader in which the point masses toggle between three colors for the three states. We implemented a custom shader using barycentric coordinates to interpolate the color based on the amount of heat, giving us a smooth and continuous mapping from heat to color. We also did not follow the discretized 2D heat propagation equation exactly, as we found the differences between outer point masses’ temperatures and the current point mass’s temperature to cause the system to explode negatively. We messed around with the constants, but ultimately settled on deleting the -4ui, j term, which greatly helped our system to stabilize. The reference paper also utilized this value called grammage, which we didn’t use because HW 4 already had us calculate a mass value for each point mass.

Implementing Fiber Fracturing and Remeshing

Disclaimer, this feature proved too difficult for us to produce, but we did try very hard. When paper has been burnt and charred, it loses the structural integrity of unburned paper. If a bending spring is stretched beyond 101.2% of its rest length, it should reorganize/delete/create its underlying spring and point mass connections. To help with this change, we have a 2D array mapping each point mass to its springs. Vector algebra is used to facilitate the reorganization. Fracturing can create and delete point masses and springs, This process invalidates some of the triangles making up the render mesh but also requires the creation of some new triangles.

Tension Mapping

This was initially created to help with fracturing. We figured it is also visually appealing, so we made it into a shader. Every spring has a rest_length and a max length of 1.1 * rest_length as dictated by Verlet Integration. We found the excess actual length as a percentage of the allowable extra length and passed that value to the shader for rendering. Although not physically correct, it’s close enough to represent actual tension forces.

Problems Encountered

Unstability of Forward Method

Using forward methods to solve finite element problems often lead to instability due to rapidly magnified extremities in later time steps, especially when dealing with small constants. Debugging and adjusting these constants can be tedious, so we focused on monitoring temperature changes at specific point masses over a few time steps and printing out temperature values for analysis. Pausing the simulation allowed us to observe point mass responses and shader displays. Introducing a counter variable to limit temperature input revealed underlying issues with the time-stepping calculation. Remeshing efforts were hampered by limitations in triangle, point mass, and spring data structures, which were primarily implemented as vectors and were inefficient for insertion and deletion operations due to pointer incompatibility with moving data. This resulted in time-consuming O(n) searches for each triangle, making efficient deletion or insertion challenging.

Takeaways

In terms of lessons learned, starting this project earlier definitely would have helped us. We did not quite realize how large and complex our scope was until we started getting into the details of our original idea. While stressful, we made a bunch of assumptions and big simplifications that ultimately led us to our current and more realistic project. We were quite optimistic in exactly how much we could leverage the provided HW 4 code and manipulate it to create more complicated objects. Our team also realized how important it is to set exact working times and come together to collaborate, rather than try to code separately and pass off the code. Our project components aren’t modular enough for fully independent work to be done, particularly in the early stages of project conception. It was helpful then for us to bounce ideas off each other in the same room and be on the same page in terms of direction moving forward.
